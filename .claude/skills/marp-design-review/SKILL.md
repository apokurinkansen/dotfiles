# Marp Design Review

Marp の Markdown プレゼン資料を PDF に変換し、視覚的にデザインをチェックして、ガイドライン違反を自動修正するスキル。

## なぜこのスキルが必要か

Marp はテキストのオーバーフロー自動制御を持たない。スライドに情報を詰め込みすぎると、テキストやテーブルがスライド外にはみ出すが、Markdown 編集中はそれに気づけない。PDF に変換して初めて視覚的な問題が判明するため、「変換 → 視覚チェック → 修正」のサイクルを自動化する。

## ワークフロー

```
Step 1: PDF 変換
Step 2: 視覚チェック（PDF を画像として読み込み）
Step 3: レビュー結果の報告
Step 4: 修正（structured-workflow で plan.md → 修正実行）
Step 5: 再変換・再チェック（問題が解消されるまで繰り返す）
```

### Step 0: テーマの確認

対象の `.md` ファイルの frontmatter を確認し、カスタムテーマが設定されているかチェックする。

- カスタムテーマがない場合: プロジェクトに `theme.css` が存在するか確認する。なければ後述の「カスタムテーマ CSS」セクションを参考に作成を提案する
- カスタムテーマがある場合: そのテーマの内容を読み込み、利用可能なクラスを把握する

### Step 1: PDF 変換

対象の `.md` ファイルを PDF に変換する。

```bash
# カスタムテーマがある場合
npx @marp-team/marp-cli <input.md> --pdf --allow-local-files --theme <theme.css> -o <output.pdf>

# カスタムテーマがない場合
npx @marp-team/marp-cli <input.md> --pdf --allow-local-files -o <output.pdf>
```

`--allow-local-files` はローカル画像の読み込みに必要。これがないと画像が壊れた表示になる。

出力先は `.claude/tmp/YYYYMMDD-marp-review/` に保存する（`date +%Y%m%d` で日付取得）。

**重要:** `.md` ファイルのディレクトリで実行すること（画像の相対パスを解決するため）。

```bash
cd <input.md のディレクトリ> && npx @marp-team/marp-cli ...
```

### Step 2: 視覚チェック

PDF を画像に変換してから読み込む。Read ツールで PDF を直接読める場合はそれでもよいが、`pdftoppm` で PNG に変換するのが確実。

```bash
# PDF を PNG 画像に変換（poppler が必要: brew install poppler）
pdftoppm -png -r 150 <output.pdf> <output-dir>/slide
# → slide-01.png, slide-02.png, ... が生成される
```

生成された画像を Read ツールで読み込み、以下のデザインガイドラインに照らしてチェックする。**すべてのスライドを確認すること。** 10枚ずつ並列で読み込むと効率的。

チェック観点:
1. **はみ出し**: テキスト、テーブル、画像がスライド領域からはみ出していないか
2. **情報量**: 1スライドに詰め込みすぎていないか
3. **余白バランス**: 上下左右に十分な余白があるか、偏りがないか
4. **見出し階層**: 不適切な見出しレベルが使われていないか
5. **テーブルサイズ**: テーブルが大きすぎてスライドに収まっていないか
6. **可読性**: フォントが小さすぎないか、コントラストは十分か
7. **画像の表示**: 画像が壊れた表示（alt テキストだけ表示）になっていないか
8. **見出しの統一性**: スライド間で見出しレベルが統一されているか（h1 と h2 の混在など）

**未作成スライド（タイトルだけで中身が空）はデザインレビューの対象外。** レビュー結果で「未作成」と記載するだけでよい。

### Step 3: レビュー結果の報告

問題のあるスライドを一覧で報告する。

```
## レビュー結果

### スライド 3: [スライドタイトル]
- ❌ テキストが下部からはみ出している
- ⚠️ 箇条書きが8項目あり、情報量が多い

### スライド 7: [スライドタイトル]
- ❌ テーブルが右側にはみ出している

### 問題なし: スライド 1, 2, 4, 5, 6, 8
```

問題がなければその旨を報告して終了する。

### Step 4: 修正

問題がある場合、structured-workflow のフローに沿って修正する。

1. レビュー結果を `.claude/tmp/YYYYMMDD-marp-review/plan.md` に落とし込む
2. plan.md にはスライドごとの修正方針を記載する（例: 「スライド3を2枚に分割」「テーブルの列を絞る」）
3. ユーザーに plan.md を提示し、アノテーションサイクルで修正方針を確定する
4. 承認後、`.md` ファイルを修正する

### Step 5: 再チェック

修正後、再度 PDF 変換 → 視覚チェックを行い、問題が解消されたか確認する。問題が残っていれば Step 3 に戻る。

## デザインガイドライン

### 1スライドの情報量

- テキスト量: **100〜150文字以内**（日本語）を目安にする
- 箇条書き: **3〜5項目が理想、最大7項目**。超える場合はスライドを分割する
- ネストの深さ: **2階層まで**。3階層以上のネストは情報の整理が不十分なサイン
- 原則「**1スライド1メッセージ**」。複数の論点を1枚に詰め込まない

### 見出し

- **h1**: スライドタイトル（各スライドに1つ）
- **h2**: サブタイトル・セクション見出し
- **h3**: 小見出し（補足的な区分）
- **h4〜h6 は使わない**: フォントサイズが本文とほぼ同じになり、視覚的に区別がつかない。h4 以降を使っている場合は、スライド分割や構成見直しで対応する
- タイトルが長い場合は `<!-- fit -->` でフィッティングを検討する

### テーブル

- **列数**: 5列以下を推奨。超える場合は重要な列だけに絞る
- **行数**: 5行以下を推奨。超える場合はスライドを分割する
- テーブルの内容が多い場合の対処法（優先順で）:
  1. 不要な列・行を削る
  2. スライドを分割する
  3. テーブル用に `<style>table { font-size: 0.7em; }</style>` でフォントを縮小する（最終手段）

### コードブロック

- **15〜20行以内** にする
- 長いコードは重要部分だけ抜粋し、全体は別途共有する
- コードブロックには Marp の auto-scaling 機能があるため、多少の超過は許容される

### 画像

- サイズ指定がない画像がスライドを圧迫していないか確認する
- 画像サイズは `![w:500](image.jpg)` のように明示指定するのが安全
- スライドの半分以上を画像に使う場合は `![bg left:50%](image.jpg)` 等の背景画像構文を使う

### 余白・レイアウト

- スライドの四辺に十分な余白があること（テーマのデフォルト padding を尊重する）
- テキストやテーブルがスライドの端ギリギリまで来ていたら情報量を減らす
- 上半分だけ、下半分だけに偏っていないか確認する

### 可読性

- 本文フォントサイズが極端に小さくなっていないか（テーマのデフォルト以下になっていないか）
- 背景色と文字色のコントラストが十分か
- 太字・色分け等の強調は1スライドに2〜3箇所まで。多用すると逆に読みにくくなる

## スタイル設定方式

よく使うパターンは**カスタムテーマ CSS の class** で定義し、一時的な微調整は **Marp ディレクティブ**で対応する。

### カスタムテーマ CSS

プロジェクトのルートに `theme.css` を作成し、md ファイルの frontmatter で指定する。

```yaml
---
marp: true
theme: custom-theme
paginate: true
---
```

テーマ CSS のベーステンプレート:

```css
/* @theme custom-theme */
@import 'default';

/* --- フォントサイズ調整クラス --- */

/* 情報量が多いスライド向け（テーブル、詳細説明等） */
section.small {
  font-size: 0.8em;
}

/* 控えめに縮小（少しだけ収まらない時） */
section.compact {
  font-size: 0.9em;
}

/* タイトルスライド向け（中央寄せ・大きめ） */
section.title {
  display: flex;
  flex-direction: column;
  justify-content: center;
  align-items: center;
  text-align: center;
}

/* --- 文字色クラス --- */

/* 強調（赤系） */
section.accent {
  --color-highlight: #e74c3c;
}
section.accent strong {
  color: var(--color-highlight);
}

/* 落ち着いたトーン（グレー系、補足スライド向け） */
section.muted {
  color: #555;
}

/* --- テーブル専用 --- */

/* テーブルのフォントだけ縮小 */
section.small-table table {
  font-size: 0.7em;
}
```

md ファイルでの使い方:

```markdown
---
<!-- _class: title -->
# プレゼンタイトル

---
<!-- _class: small -->
# テーブルが多いスライド
| 列1 | 列2 | 列3 |
|-----|-----|-----|

---
<!-- _class: accent -->
# 重要なポイント
**ここが強調される**
```

### Marp ディレクティブ（一時的な微調整）

スライドごとに個別でスタイルを変えたい場合、ディレクティブを使う。カスタムテーマに class を増やすほどでもない一回限りの調整に向いている。

```markdown
---
<!-- _color: #333 -->
<!-- _backgroundColor: #f5f5f5 -->
# このスライドだけ背景をグレーに
```

主なディレクティブ:
- `<!-- _color: ... -->` — 文字色
- `<!-- _backgroundColor: ... -->` — 背景色
- `<!-- _class: ... -->` — テーマの CSS クラスを適用
- `<!-- _header: ... -->` — ヘッダーテキスト
- `<!-- _footer: ... -->` — フッターテキスト
- `<!-- _paginate: false -->` — ページ番号を非表示

### 修正時のスタイル適用方針

Step 4 の修正で、スタイル調整が必要な場合は以下の優先順で対応する:

1. **コンテンツを減らす**（情報量の問題は CSS ではなく内容で解決する）
2. **カスタムテーマの class を適用する**（`small`, `compact`, `small-table` 等）
3. **Marp ディレクティブで個別調整する**（そのスライド固有の色・背景等）
4. **カスタムテーマに新しい class を追加する**（同じパターンが3回以上出現する場合）
